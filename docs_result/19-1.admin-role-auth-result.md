# 18. Admin 인증 방식 교체 — x-admin-key → 세션 + 역할 기반 구현 결과

> 작성일: 2026-03-03
> 대상 패키지: `apps/api`

---

## 1. 작업 범위 요약

| 항목 | 내용 |
|---|---|
| **변경 대상** | `GET /api/admin/analytics` 인증 로직 |
| **기존 방식** | `x-admin-key` 헤더 vs `env.ADMIN_SECRET` 비교 |
| **신규 방식** | 세션 쿠키(`aso_session`) + D1 `user_auth_profiles.role` 컬럼 조회 |
| **사전 조건** | `sessionCookieMiddleware`가 `/api/*`에 이미 적용되어 있음 |

---

## 2. 배경 및 동기

기존 `x-admin-key` 방식은 다음 문제를 가집니다:

- 헤더 값이 브라우저 Network 탭에 노출됨 (`NEXT_PUBLIC_ADMIN_KEY`)
- 세션과 독립적으로 관리되어 로그아웃 시 무효화 불가
- 단순 문자열 비교로 타이밍 공격에 취약

신규 방식은 기존 세션 인프라 위에 역할(role) 컬럼만 추가하여 **서버 측 DB 권한 체크**로 전환합니다.

## 3. 변경 파일

### 3.1 `apps/api/src/routes/admin.analytics.ts` (전체)

```typescript
import { Hono } from 'hono'
import type { AppEnv } from '../env'

const adminRouter = new Hono<AppEnv>()

adminRouter.get('/api/admin/analytics', async (c) => {
  // --- Auth ---
  const sessionUid = c.get('sessionUid')

  if (!sessionUid) {
    return c.json({ ok: false, error: 'Unauthorized' }, 401)
  }

  const db = c.env.DB

  const profile = await db
    .prepare(`SELECT role FROM user_auth_profiles WHERE uid = ?`)
    .bind(sessionUid)
    .first<{ role: string }>()

  if (!profile || profile.role !== 'admin') {
    return c.json({ ok: false, error: 'Forbidden' }, 403)
  }

  // --- KPI counts ---
  const kpiRows = await db
    .prepare(
      `SELECT event_name, COUNT(*) as count
       FROM events
       WHERE event_name IN (
         'generate_start','generate_success','limit_exceeded',
         'result_view','upgrade_click'
       )
       GROUP BY event_name`,
    )
    .all<{ event_name: string; count: number }>()

  const kpi: Record<string, number> = {}
  for (const row of kpiRows.results) {
    kpi[row.event_name] = row.count
  }

  const starts    = kpi['generate_start']   ?? 0
  const successes = kpi['generate_success'] ?? 0
  const limits    = kpi['limit_exceeded']   ?? 0
  const views     = kpi['result_view']      ?? 0
  const upgrades  = kpi['upgrade_click']    ?? 0

  const pct = (num: number, den: number) =>
    den === 0 ? null : Math.round((num / den) * 1000) / 10

  // --- Plan distribution ---
  const planRows = await db
    .prepare(
      `SELECT user_state, COUNT(*) as count
       FROM events
       WHERE event_name = 'generate_start'
       GROUP BY user_state`,
    )
    .all<{ user_state: string; count: number }>()

  // --- Hourly (last 24 h) ---
  const hourlyRows = await db
    .prepare(
      `SELECT substr(created_at, 1, 13) as hour, COUNT(*) as count
       FROM events
       WHERE event_name = 'generate_start'
         AND created_at >= datetime('now', '-1 day')
       GROUP BY hour
       ORDER BY hour ASC`,
    )
    .all<{ hour: string; count: number }>()

  return c.json({
    ok: true,
    kpi: {
      generate_start:   starts,
      generate_success: successes,
      limit_exceeded:   limits,
      result_view:      views,
      upgrade_click:    upgrades,
      success_rate:     pct(successes, starts),
      limit_rate:       pct(limits, successes),
      upgrade_rate:     pct(upgrades, limits),
    },
    plan_distribution: planRows.results,
    hourly:            hourlyRows.results,
  })
})

export default adminRouter
```

## 4. 인증 흐름 상세

```
클라이언트 요청
GET /api/admin/analytics
Cookie: aso_session=<rawToken>
         │
         ▼
sessionCookieMiddleware (apps/api/src/middleware/session-cookie.ts)
  └─ SHA-256(rawToken) → sessions 테이블 조회
  └─ 유효한 세션이면 c.set('sessionUid', uid)
         │
         ▼
adminRouter.get('/api/admin/analytics')
  └─ sessionUid 없음? → 401 Unauthorized
  └─ SELECT role FROM user_auth_profiles WHERE uid = sessionUid
  └─ profile 없거나 role ≠ 'admin'? → 403 Forbidden
  └─ role = 'admin' → D1 analytics 쿼리 실행 → 200 OK
```

## 5. 인증 응답 매트릭스

| 상태 | 응답 코드 | 응답 바디 |
|---|---|---|
| `aso_session` 쿠키 없음 | 401 | `{ ok: false, error: 'Unauthorized' }` |
| 세션 만료 또는 폐기됨 | 401 | `{ ok: false, error: 'Unauthorized' }` |
| 세션 유효, `role = 'user'` | 403 | `{ ok: false, error: 'Forbidden' }` |
| `user_auth_profiles` 행 없음 | 403 | `{ ok: false, error: 'Forbidden' }` |
| 세션 유효, `role = 'admin'` | 200 | analytics JSON |

## 6. role 컬럼 선행 조건

이 변경이 동작하려면 `user_auth_profiles` 테이블에 `role` 컬럼이 존재해야 합니다.
마이그레이션은 **`20-1.production-hardening-dashboard-result.md`** 의 migration 0004에서 추가됩니다.

```sql
-- 0004_role_column.sql
ALTER TABLE user_auth_profiles
  ADD COLUMN role TEXT NOT NULL DEFAULT 'user'
    CHECK(role IN ('user', 'admin'));
```

어드민 계정 부여 방법:
```sql
UPDATE user_auth_profiles
SET role = 'admin'
WHERE email_normalized = 'admin@example.com';
```

## 7. 기존 방식과의 비교

| 항목 | 이전 (x-admin-key) | 현재 (세션 + role) |
|---|---|---|
| 인증 매체 | HTTP 헤더 | HTTP-only 쿠키 |
| 권한 저장 위치 | 환경 변수 | D1 `user_auth_profiles.role` |
| 로그아웃 시 무효화 | 불가 | 가능 (`sessions.revoked_at`) |
| 브라우저 노출 | NEXT_PUBLIC 변수로 노출됨 | 없음 |
| 권한 변경 반영 | 재배포 필요 | SQL UPDATE 즉시 반영 |
| 다중 어드민 지원 | 단일 시크릿 공유 | 개별 계정별 독립 |

## 9. 보안 고려 사항

- `ADMIN_SECRET` 환경 변수 제거 가능 — 더 이상 사용되지 않음
- 세션 토큰은 SHA-256 해시로만 DB에 저장 → 탈취 시에도 원본 토큰 역산 불가
- `role` 컬럼은 `DEFAULT 'user'` — 신규 가입자는 자동으로 일반 사용자
- 어드민 승격은 DB 직접 수정으로만 가능 → API를 통한 자동 승격 경로 없음
