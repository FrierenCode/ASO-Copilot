## 1) 설계 문서

### 1-1. Production-safe 원칙
1. 기존 `users`, `usage_monthly`, `user_entitlements` 구조는 유지한다.
2. 마이그레이션은 **additive only**로 신규 테이블/인덱스만 추가한다.
3. 기존 Polar webhook entitlement 흐름은 그대로 두고, v2 usage 정책은 별도 gate 계층에서 처리한다.
4. anonymous도 항상 `users` row를 갖게 유지한다(기존 uid-cookie 미들웨어 upsert 활용).
5. 고트래픽 카운터 테이블은 FK를 최소화해 runtime FK 실패를 방지한다.

### 1-2. Anonymous -> Member 승격 흐름 (UID 유지)
1. 최초 방문:
- `aso_uid` 없으면 발급/서명 쿠키 설정
- `users` upsert
- `user_auth_profiles`에 `uid` 기준 row upsert (`user_type='anonymous'`)
2. Magic Link 검증 성공:
- 현재 uid-cookie의 uid를 canonical 후보로 사용
- 해당 이메일이 미등록이면 같은 uid를 `member`로 승격 (`uid` 유지)
- 해당 이메일이 이미 다른 uid에 등록되어 있으면 기존 member uid로 세션 발급(결제/entitlement 일관성 우선)
3. `subscription.metadata.reference_id` 기반 매핑은 `uid`를 키로 계속 동작하므로 깨지지 않는다.

### 1-3. Entitlement 우선순위
1. `user_entitlements`에 `plan_code='pro' AND status='active'`이면 무조건 무제한.
2. 위 조건이 아니고 로그인(session 유효)이면 Free monthly 5.
3. 로그인 안 했으면 anonymous lifetime 2.
4. 즉, `user_entitlements`는 Pro 판정의 단일 진실원천으로 유지하고, 비Pro quota는 auth 상태로 분기한다.

### 1-4. Rate gate 로직 (기본: postsuccess + idempotency)
1. `x-request-id` 필수.
2. `(uid, request_id)` idempotency claim 생성(`generation_idempotency`).
3. 사전 한도 검사(precheck):
- Pro: 통과
- Member: `usage_monthly`에서 이번 달 사용량 < 5인지 확인
- Anonymous: `usage_lifetime` < 2인지 확인
4. generate 실행.
5. 성공 후에만 원자적 차감(postsuccess):
- Member: `usage_monthly` 조건부 증가 `used_count < 5`
- Anonymous: `usage_lifetime` 조건부 증가 `used_count < 2`
6. 차감 성공 시 `generation_requests`에 최종 allowed 기록.
7. 차감 실패(경합) 시 429 반환(생성 결과 폐기), `generation_idempotency.status='limit_exceeded'`.

### 1-5. 쿠키 정책
1. `aso_uid`: `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`, `Max-Age=31536000`.
2. `aso_session`: `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`, `Max-Age=2592000`.
3. 서명/토큰은 Web Crypto 기반(HMAC/SHA-256), 평문 저장 금지.
4. CORS는 origin allowlist + credentials true 조합으로만 허용.

### 1-6. Session 정책
1. 세션은 DB `sessions`에 `session_token_hash`로 저장(원문 토큰 미저장).
2. absolute expiry(예: 30일) + 선택적 sliding touch.
3. logout 시 `revoked_at` 세팅 + 쿠키 제거.
4. session middleware는 webhook 경로에는 적용하지 않음.

### 1-7. API 스펙 (최종)

#### `POST /auth/request-magic-link`
Request:
```json
{
  "email": "user@example.com",
  "redirect_to": "/"
}
```
Response `202`:
```json
{ "ok": true }
```
Errors:
- `400 INVALID_EMAIL`
- `429 MAGIC_LINK_RATE_LIMITED`
- `500 EMAIL_SEND_FAILED`

#### `GET /auth/verify?token=...&redirect_to=...`
- 성공: `302` redirect + `aso_session` 설정 + 필요시 `aso_uid` 동기화
- 실패:
  - `400 INVALID_TOKEN`
  - `410 MAGIC_LINK_EXPIRED_OR_USED`

#### `POST /auth/logout`
Response `200`:
```json
{ "ok": true }
```

#### `GET /api/me`
Response `200` 예시:
```json
{
  "uid": "uuid",
  "authenticated": true,
  "plan": "pro | free | anonymous",
  "usage": {
    "policy": "unlimited | monthly | lifetime",
    "limit": null,
    "used": 0,
    "remaining": null,
    "reset_at": null
  }
}
```

#### `POST /generate` (v2 gate)
Headers:
- `x-request-id` (required)

429 응답(요구 형식):
```json
{
  "ok": false,
  "error": "LIMIT_EXCEEDED",
  "plan": "anonymous | free",
  "upgrade_url": "/pricing"
}
```

추가 에러:
- `400 MISSING_IDEMPOTENCY_KEY`
- `409 DUPLICATE_REQUEST` (이미 처리 중/처리 완료)

### 1-9. 동시성 처리 전략
1. idempotency key: `x-request-id` 필수.
2. `generation_idempotency`에서 `(uid, request_id)` PK로 선점(claim).
3. `generation_requests`는 기존 `(uid, request_id)` PK를 최종 outcome audit로 유지.
4. race 방지:
- 카운터 증가는 `UPDATE ... WHERE used_count < limit` 조건부 원자 증가
- 변경 row 수 0이면 초과로 처리
- precheck + postsuccess의 이중 보호
5. 중복 클릭:
- 동일 `x-request-id` 재요청은 `409` 또는 직전 결과 재응답 정책 중 택1(권장: 409 + 클라이언트 재시도 방지)

### 1-10. 운영 체크리스트
1. Staging에서 anonymous 2회/3회차 429 검증
2. Staging에서 member 5회/6회차 429 검증
3. Pro entitlement 회귀: webhook 후 즉시 무제한 확인
4. Magic link: 발급/만료/1회사용/로그아웃 검증
5. 쿠키 보안: `HttpOnly/Secure/SameSite` 헤더 검증
6. CORS: allowlist origin + credentials 검증
7. `/generate` 429 body가 요구 JSON 구조와 일치하는지 검증
8. 장애 시 feature flag로 v1 gate 즉시 복귀 가능 여부 확인

## 2) Migration SQL (Safe Version)

```sql
PRAGMA foreign_keys = ON;

-- 1) 사용자 auth/profile 상태 (users 테이블 변경 없이 확장)
CREATE TABLE IF NOT EXISTS user_auth_profiles (
  uid TEXT PRIMARY KEY,
  user_type TEXT NOT NULL DEFAULT 'anonymous'
    CHECK(user_type IN ('anonymous','member')),
  email TEXT,
  email_normalized TEXT,
  email_verified_at TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_user_auth_profiles_email_norm_unique
  ON user_auth_profiles(email_normalized)
  WHERE email_normalized IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_user_auth_profiles_user_type
  ON user_auth_profiles(user_type);

-- 2) Magic Link 토큰 저장 (원문 토큰 저장 금지)
CREATE TABLE IF NOT EXISTS auth_magic_links (
  magic_link_id TEXT PRIMARY KEY,
  email_normalized TEXT NOT NULL,
  token_hash TEXT NOT NULL UNIQUE,
  requested_uid TEXT,
  target_uid TEXT,
  redirect_to TEXT,
  expires_at TEXT NOT NULL,
  consumed_at TEXT,
  invalidated_at TEXT,
  requested_ip_hash TEXT,
  requested_ua_hash TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE INDEX IF NOT EXISTS idx_auth_magic_links_email_created
  ON auth_magic_links(email_normalized, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_auth_magic_links_expires_consumed
  ON auth_magic_links(expires_at, consumed_at);

-- 3) 세션 저장 (원문 세션 토큰 저장 금지)
CREATE TABLE IF NOT EXISTS sessions (
  session_id TEXT PRIMARY KEY,
  session_token_hash TEXT NOT NULL UNIQUE,
  uid TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active'
    CHECK(status IN ('active','revoked','expired')),
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  last_seen_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  expires_at TEXT NOT NULL,
  revoked_at TEXT,
  ip_hash TEXT,
  user_agent_hash TEXT
);

CREATE INDEX IF NOT EXISTS idx_sessions_uid_status
  ON sessions(uid, status);

CREATE INDEX IF NOT EXISTS idx_sessions_expires_status
  ON sessions(expires_at, status);

-- 4) anonymous lifetime usage (FK 없음: high-write 경로 FK 실패 방지)
CREATE TABLE IF NOT EXISTS usage_lifetime (
  uid TEXT PRIMARY KEY,
  used_count INTEGER NOT NULL DEFAULT 0 CHECK(used_count >= 0),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE INDEX IF NOT EXISTS idx_usage_lifetime_updated
  ON usage_lifetime(updated_at);

-- 5) postsuccess idempotency claim 테이블
CREATE TABLE IF NOT EXISTS generation_idempotency (
  uid TEXT NOT NULL,
  request_id TEXT NOT NULL,
  status TEXT NOT NULL
    CHECK(status IN ('processing','succeeded','limit_exceeded','failed')),
  plan_at_request TEXT,
  error_code TEXT,
  started_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  finished_at TEXT,
  PRIMARY KEY (uid, request_id)
);

CREATE INDEX IF NOT EXISTS idx_generation_idempotency_status_started
  ON generation_idempotency(status, started_at);

-- 6) 기존 generation_requests 보조 인덱스 (테이블 구조 변경 없음)
CREATE INDEX IF NOT EXISTS idx_generation_requests_created_at
  ON generation_requests(created_at);

CREATE INDEX IF NOT EXISTS idx_generation_requests_plan_decision
  ON generation_requests(plan_code, decision);
```

## 4) 리스크 분석

1. 이메일-UID 충돌 리스크  
- 위험: 기존 이메일이 다른 uid에 이미 연결된 경우 승격 충돌  
- 대응: 기존 이메일 uid 우선 로그인, 자동 merge 금지(수동 merge 별도)

2. postsuccess 경합 리스크  
- 위험: 동시 요청에서 생성 후 차감 실패로 결과 폐기 발생  
- 대응: precheck + 조건부 update + 명확한 429 처리, 향후 reservation 옵션 도입

3. idempotency 누락 리스크  
- 위험: 클라이언트가 `x-request-id` 누락 시 중복 과금/중복 생성  
- 대응: 헤더 필수화(`400 MISSING_IDEMPOTENCY_KEY`)

4. 쿠키/CORS 오배치 리스크  
- 위험: 세션 쿠키 미전송으로 로그인 상태 인식 실패  
- 대응: `credentials: include`, 명시 origin allowlist, `Secure/SameSite` 검증 자동화

5. entitlement 회귀 리스크  
- 위험: v2 gate가 pro 우선 판정을 누락하면 유료 사용자 차단  
- 대응: gate 첫 단계에서 `pro active` 최우선 판정 + webhook 회귀 테스트 필수

6. FK/데이터 무결성 리스크  
- 위험: high-write usage 경로에서 FK 실패  
- 대응: `usage_lifetime`, `generation_idempotency`는 FK 제거, `users` upsert를 미들웨어에서 강제
