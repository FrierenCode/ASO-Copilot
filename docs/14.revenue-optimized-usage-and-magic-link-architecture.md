## 1) 설계 문서 (Markdown)

### 목표
- 비회원(anonymous): **누적 2회 무료** (월 리셋 없음)
- 회원 Free: **월 5회**
- Pro: **무제한** (`user_entitlements.plan_code='pro' AND status='active'`)
- Magic Link 로그인 도입 (세션 쿠키 기반)
- 기존 Polar webhook + entitlements 흐름 유지

### A. 시스템 설계

#### A-1. 요청 흐름 (/generate)
1. `uid` 쿠키 미들웨어가 `aso_uid`(서명) 검증/발급.
2. 세션 미들웨어가 `aso_session` 검증.
3. Identity 결정:
   - 세션 유효 시 `effective_uid = session.uid` (로그인 사용자)
   - 세션 없으면 `effective_uid = uid_cookie` (anonymous)
4. Plan/한도 결정 우선순위:
   - 1순위: `user_entitlements`에서 `pro active`면 무제한
   - 2순위: 로그인 사용자면 Free 월 5회
   - 3순위: anonymous면 lifetime 2회
5. Quota gate 통과 시 generate 실행.
6. 실패 시 `429 LIMIT_EXCEEDED` + `upgrade_url="/pricing"`.

#### A-2. 인증/세션 흐름 (Magic Link)
1. `POST /auth/request-magic-link`로 이메일 입력.
2. 서버는 토큰 원문을 생성하고 DB에는 **hash만 저장**.
3. 이메일 벤더 추상화 인터페이스로 링크 전송 (`/auth/verify?token=...`).
4. `GET /auth/verify`에서 토큰 hash 검증 + 만료/재사용 검사.
5. 성공 시 세션 생성:
   - `sessions` row 생성
   - `aso_session` HttpOnly 쿠키 설정
   - `aso_uid`를 계정의 canonical uid로 동기화
6. `POST /auth/logout`에서 세션 revoke + 쿠키 삭제.

#### A-3. 쿠키/보안 정책
1. `aso_uid`: `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`, 1년.
2. `aso_session`: `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`, 30일(슬라이딩 연장 가능).
3. magic link token/session token은 DB에 평문 저장 금지 (SHA-256 hash 저장).
4. `/auth/verify` redirect 파라미터는 allowlist 경로만 허용.
5. CORS는 명시 origin + credentials 허용.
6. entitlements/plan 판단은 서버만 신뢰, 클라이언트 저장소 사용 금지.

#### A-4. Rate limit 설계
1. `/auth/request-magic-link`:
   - IP 기준 제한 (예: 10회/15분)
   - email 기준 제한 (예: 3회/15분)
2. `/generate`:
   - 정책 quota + 인프라 rate limiting(WAF/Rules) 병행.
3. 과금 보호 우선: quota가 먼저 차단, 그 다음 모델 호출.

#### A-5. Entitlement 우선순위 (핵심 규칙)
1. `pro active`면 무조건 무제한.
2. pro가 아니고 로그인 상태면 Free 월 5회.
3. 로그인 상태가 아니면 anonymous lifetime 2회.
4. 기존 Polar webhook는 계속 `user_entitlements`를 업데이트하고, 이 값이 최상위 판단 소스.

### C. API 스펙

#### C-1. `POST /auth/request-magic-link`
- Request:
```json
{
  "email": "user@example.com",
  "redirect_to": "/"
}
```
- Response `202`:
```json
{ "ok": true }
```
- Error:
  - `400 INVALID_EMAIL`
  - `429 MAGIC_LINK_RATE_LIMITED`
  - `500 EMAIL_SEND_FAILED`

#### C-2. `GET /auth/verify?token=...&redirect_to=...`
- 성공: `302` 리다이렉트 + `aso_session`, `aso_uid` 쿠키 설정
- 실패:
  - `400 INVALID_TOKEN`
  - `410 MAGIC_LINK_EXPIRED_OR_USED`

#### C-3. `POST /auth/logout`
- Response `200`:
```json
{ "ok": true }
```
- 동작: 세션 revoke + `aso_session` 쿠키 만료

#### C-4. `GET /api/me`
- Response `200`:
```json
{
  "uid": "uuid",
  "authenticated": true,
  "email": "user@example.com",
  "plan": "free",
  "usage": {
    "policy": "monthly",
    "limit": 5,
    "used": 2,
    "remaining": 3,
    "reset_at": "2026-04-01T00:00:00Z"
  }
}
```
- plan 값: `anonymous | free | pro`
- `reset_at`: anonymous/pro는 `null`

#### C-5. `POST /generate` (기존 + rate gate 강화)
- 성공 `200`: 기존 generate 응답
- 실패 `429`:
```json
{
  "ok": false,
  "error": "LIMIT_EXCEEDED",
  "upgrade_url": "/pricing"
}
```
- 실패 `409`:
```json
{
  "ok": false,
  "error": "DUPLICATE_REQUEST"
}
```

### E. 동시성/중복 클릭 대비 전략 (2안)

#### 최소안 (MVP): 요청 시작 시 차감
1. idempotency key(`x-request-id`)를 `generation_requests(uid, request_id)`로 dedupe.
2. 요청 시작 시 quota 원자적 증가.
3. 증가 실패 시 즉시 429.
4. 장점: 단순, 초과 사용 방지 강함.
5. 단점: 생성 실패/네트워크 실패도 차감될 수 있음.

#### 권장안: 성공 시 차감 + idempotency key
1. 요청 시작 시 `generation_requests`에 `started` 기록(중복 방지).
2. 생성 성공 후 quota 차감 시도.
3. 차감 성공 시 `allowed+counted=1`로 확정.
4. 차감 실패 시 `rejected_limit`로 기록하고 429 반환.
5. 장점: 과금 공정성 높음(성공 요청만 차감).
6. 단점: 동시성 높은 상황에서 계산 낭비 가능.
7. 보완: 권장안에 “slot reservation(짧은 TTL)”을 추가하면 낭비를 줄일 수 있음.

### F. 검증 시나리오 체크리스트

#### 로컬
1. anonymous 1~2회 성공, 3회차 429 확인.
2. 월이 바뀌어도 anonymous는 계속 누적 제한 유지 확인.
3. magic link 요청/검증/로그아웃 기본 동작 확인.
4. member free는 월 5회, 6회차 429 확인.
5. pro entitlement 주입 시 무제한 확인.

#### 스테이징
1. 실제 이메일 전송 벤더 연결 검증.
2. 토큰 만료(예: 15분) 및 1회 사용 제한 확인.
3. 동시 요청(더블클릭/병렬 탭)에서 idempotency 동작 확인.
4. webhook로 pro 반영 후 `/api/me`, `/generate` 즉시 반영 확인.

#### 프로덕션
1. 쿠키 속성(`HttpOnly/Secure/SameSite`) 헤더 검증.
2. CORS credentials + origin whitelist 검증.
3. `/auth/request-magic-link` abuse 차단(429 비율 모니터링).
4. `uid_not_resolved`, `LIMIT_EXCEEDED`, `MAGIC_LINK_EXPIRED` 지표 모니터링.
5. 기존 Polar entitlement 업데이트 흐름 회귀 확인.

## 2) D1 Migration SQL

```sql
PRAGMA foreign_keys = OFF;
BEGIN TRANSACTION;

-- ---------------------------------------------------------------------------
-- 1) users 확장: anonymous/member + email 로그인 지원
-- ---------------------------------------------------------------------------
ALTER TABLE users ADD COLUMN email TEXT;
ALTER TABLE users ADD COLUMN email_normalized TEXT;
ALTER TABLE users ADD COLUMN user_type TEXT NOT NULL DEFAULT 'anonymous'
  CHECK (user_type IN ('anonymous','member'));
ALTER TABLE users ADD COLUMN email_verified_at TEXT;
ALTER TABLE users ADD COLUMN updated_at TEXT;

UPDATE users
SET updated_at = COALESCE(updated_at, last_seen_at, strftime('%Y-%m-%dT%H:%M:%fZ','now'));

CREATE UNIQUE INDEX IF NOT EXISTS idx_users_email_normalized_unique
  ON users(email_normalized)
  WHERE email_normalized IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_users_user_type
  ON users(user_type);

-- ---------------------------------------------------------------------------
-- 2) magic link 토큰 저장 (원문 저장 금지: token_hash만 저장)
-- ---------------------------------------------------------------------------
CREATE TABLE IF NOT EXISTS auth_magic_links (
  magic_link_id TEXT PRIMARY KEY,
  email_normalized TEXT NOT NULL,
  token_hash TEXT NOT NULL UNIQUE,
  requested_by_uid TEXT REFERENCES users(uid) ON DELETE SET NULL,
  target_uid TEXT REFERENCES users(uid) ON DELETE SET NULL,
  redirect_to TEXT,
  expires_at TEXT NOT NULL,
  used_at TEXT,
  invalidated_at TEXT,
  requested_ip_hash TEXT,
  requested_ua_hash TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

CREATE INDEX IF NOT EXISTS idx_magic_links_email_created
  ON auth_magic_links(email_normalized, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_magic_links_expires_used
  ON auth_magic_links(expires_at, used_at);

-- ---------------------------------------------------------------------------
-- 3) session 저장 (원문 저장 금지: session_token_hash만 저장)
-- ---------------------------------------------------------------------------
CREATE TABLE IF NOT EXISTS sessions (
  session_id TEXT PRIMARY KEY,
  session_token_hash TEXT NOT NULL UNIQUE,
  uid TEXT NOT NULL REFERENCES users(uid) ON DELETE CASCADE,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  last_seen_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  expires_at TEXT NOT NULL,
  revoked_at TEXT,
  ip_hash TEXT,
  user_agent_hash TEXT
);

CREATE INDEX IF NOT EXISTS idx_sessions_uid
  ON sessions(uid);

CREATE INDEX IF NOT EXISTS idx_sessions_expires_revoked
  ON sessions(expires_at, revoked_at);

-- ---------------------------------------------------------------------------
-- 4) usage_monthly를 (uid, yyyymm) 구조로 전환
--    기존 usage_monthly(period_start) 데이터를 집계 이관
-- ---------------------------------------------------------------------------
ALTER TABLE usage_monthly RENAME TO usage_monthly_legacy;

CREATE TABLE IF NOT EXISTS usage_monthly (
  uid TEXT NOT NULL REFERENCES users(uid) ON DELETE CASCADE,
  yyyymm TEXT NOT NULL CHECK(length(yyyymm) = 6),
  used_count INTEGER NOT NULL DEFAULT 0 CHECK(used_count >= 0),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  PRIMARY KEY (uid, yyyymm)
);

INSERT INTO usage_monthly (uid, yyyymm, used_count, updated_at)
SELECT
  uid,
  strftime('%Y%m', period_start) AS yyyymm,
  SUM(used_count) AS used_count,
  MAX(updated_at) AS updated_at
FROM usage_monthly_legacy
GROUP BY uid, strftime('%Y%m', period_start);

CREATE INDEX IF NOT EXISTS idx_usage_monthly_yyyymm
  ON usage_monthly(yyyymm);

-- ---------------------------------------------------------------------------
-- 5) anonymous lifetime usage 카운터
-- ---------------------------------------------------------------------------
CREATE TABLE IF NOT EXISTS usage_lifetime (
  uid TEXT PRIMARY KEY REFERENCES users(uid) ON DELETE CASCADE,
  used_count INTEGER NOT NULL DEFAULT 0 CHECK(used_count >= 0),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

COMMIT;
PRAGMA foreign_keys = ON;
```
