# Cloudflare Pages 배포 수정 플랜 + 최소 코드 하드닝

작성일: 2026-03-03  
프로젝트: ASO Copilot (pnpm 모노레포, `apps/web` Next.js static export)

## 배경

- 현재 Pages 설정
  - Root directory: `(empty)`
  - Build command: `pnpm build`
  - Build output directory: `apps/web/out`
- 증상: 운영에서 `/admin` 404
- API base env: `NEXT_PUBLIC_API_BASE=https://api.gigigitg95.workers.dev`

---

## A) 왜 현재 Build Command가 모노레포에서 문제를 만들 수 있는가

현재 설정도 동작 가능하지만, 모노레포에서는 아래 리스크가 큼.

1. 루트 스크립트(`pnpm build`)가 바뀌거나 실패해도 `apps/web` export가 확실히 수행되지 않을 수 있음.
2. 하위 경로 출력(`apps/web/out`)은 캐시/이전 아티팩트와 섞였을 때 디버깅이 어려움.
3. 결과적으로 `/admin`에 필요한 `out/admin/index.html`이 없거나 stale 상태여도 늦게 발견됨.

핵심은 Pages가 매 배포마다 `apps/web`를 직접 빌드하고 `out`를 정확히 수집하도록 고정하는 것.

---

## B) 올바른 Pages 설정 2안

## Option A (권장)
1. Root directory = `apps/web`
2. Build command = `pnpm build`
3. Output directory = `out`

권장 이유:
- 빌드 루트/출력 경로가 동일 컨텍스트라 실수 가능성이 가장 낮음.
- `/admin` 산출물 누락 여부를 로그와 파일 트리에서 바로 확인 가능.

## Option B
1. Root directory = `(empty)`
2. Build command = `pnpm -C apps/web build`
3. Output directory = `apps/web/out`

주의:
- 루트 기준 경로를 유지해야 하므로 설정 실수 여지가 Option A보다 큼.

## D) 코드 하드닝 요구사항 및 패치 지침

대상 파일: `apps/web/src/app/admin/page.tsx`

## 1. NEXT_PUBLIC_ADMIN_KEY 누락 시
- 크래시 금지
- 명확한 UI 메시지 출력:
  - 제목: `Admin key not configured`
  - 안내: `Set NEXT_PUBLIC_ADMIN_KEY in Cloudflare Pages environment variables, then trigger a new deploy.`

## 2. API 401 응답 시
- 명확한 UI 메시지 출력:
  - `Unauthorized: check NEXT_PUBLIC_ADMIN_KEY matches ADMIN_SECRET`

## 3. 방어 로직
1. `ADMIN_KEY` 누락이면 fetch 호출하지 않고 즉시 오류 상태 렌더링
2. `response.status === 401` 분기 처리
3. 기타 비정상 응답은 `HTTP <status>` 형태로 표시
4. 로딩/에러/정상 상태 분리 렌더링

## 4. 관련 TopNav 노출 확인
- `Login`, `Signup` 링크가 모든 주요 페이지에서 보이도록 유지
- 최소 확인 페이지:
  1. `/`
  2. `/try`
  3. `/result`
  4. `/admin`
  5. `/pricing`
